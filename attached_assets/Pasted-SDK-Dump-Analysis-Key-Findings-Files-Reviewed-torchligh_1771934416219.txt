3. 🎯 CRITICAL DISCOVERY: FightMgr.MapGamePlay at offset 0x800
The FightMgr class (found as a singleton in GObjects at /Engine/Transient) has:

Code
[Offset:0x7B0] MapProperty FightMgr:MapCustomTrap  ← TMap<int32 entity_id, EMapCustomTrap*>
[Offset:0x800] MapProperty FightMgr:MapGamePlay    ← TMap<int32 entity_id, EGameplay*>  ← !!
[Offset:0x850] MapProperty FightMgr:MapServant
[Offset:0x8A0] ObjectProperty FightMgr:LevelInfo
These two maps are completely missing from FIGHTMGR_OFFSETS in constants.py!

The MapGamePlay TMap maps int32 logic_entity_id → EGameplay*. The logic entity IDs ARE the values from our existing constants (0xD8 for Carjack, 0xDB/0xDD/0xDE for Sandlord). By reading this TMap we get:

Which entity type is active (the int32 key matches EGAMEPLAY_CARJACK_IDS or EGAMEPLAY_SANDLORD_IDS)
The exact EGameplay pointer (the value) → read its RootComponent position
The existing portal_detector.py already has a working TMap reader (_detect_tmap_layout(), strides [24, 32, 16], value offsets [0x10, 0x08, 0x18]) for FightMgr.MapPortal. The exact same infrastructure reads MapGamePlay — we just also need to read the key at offset 0 (4-byte int32 before the pointer).

4. TMap element layout for TMap<int32, EEntity*>
Code
struct TSetElement {
  int32   Key    @ offset 0   ← logic_entity_id (0xD8, 0xDB, etc.)
  int32   _pad   @ offset 4   ← alignment padding
  EEntity* Value @ offset 8   ← EGameplay* we already use
  int32   HashNext @ offset 16
  int32   HashIndex @ offset 20
}
Total stride = 24 bytes  ← already tried by portal_detector!
Value offset = 0x08      ← already tried by portal_detector!
Key offset = 0x00        ← just read an extra int32 at this position
