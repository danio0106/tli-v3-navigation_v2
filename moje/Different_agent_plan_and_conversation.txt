Objective
Find a reliable field to distinguish Carjack security guards from regular monsters, using the SDK dump (6 guards alive, nothing else) and runtime memory reading. ABP-based detection fails on maps where guard ABP overlaps with native monster ABP. Need a universal marker.

SDK Dump Analysis Summary
From the Objects/Names dump with 6 guards only:

ECfgComponent (default subobject on EMonster, accessible via ueComponents TMap): has CfgInfo.ID (int32 at offset 0x120). Each monster config has a unique ID. Guards should share one or a few IDs.
EMapIconComponent (default subobject on EMonster): Size=0x158 but NO reflected UPROPERTY fields — icon type likely stored as non-reflected C++ member. Raw byte dump needed.
EBattleCryCamp enum: E_hostile(0), E_friendly(1), E_own(2), E_minion(3) — guards might use a different camp value.
ESide enum: ESide_monster(1), ESide_player(2), ESide_monster_bomb(3), etc. — guards could have a unique Side.
QAMonsterInfo.CreateSourceType (enum at +0x4 in QAMonsterInfo struct): tells how monster was spawned.
The ueComponents TMap at EEntity+0x288 already traversed for EAnimeComponent; same path can read ECfgComponent and EMapIconComponent.
